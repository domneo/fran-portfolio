        showSectionTitle: true

          - content: |
              ### Taking complexity out of the user's hands

              Manual permission-setting is where access systems go wrong - it's the source of most user error and security gaps. I worked with a small user group to fix the rights for each archetype, then had developers hardcode them.

              This was a deliberately rigid approach to solve a problem that's easy to miss until the system won't boot.

              ### The Bootstrap Paradox

              * No Super Admin defined → the system can't propagate a chain of command.
              * No roles with owner-type rights → the modules are unusable, so the system can't run.

              **You can't use the system to set up the permissions it needs in order to be usable.** The way out was a hardcoded master logic in the backend. This provided a guaranteed baseline that makes the System operational from the moment of deployment, before anyone configures anything.

              ([Read about the full research process here](https://frn-cne.vercel.app/st-dsta/ux-process-and-research). The UX process outlined in this case study is a subset of the full process and only focuses on the Access Controls module.)

              To begin, we first engaged users of a higher level of command to obtain their chain of command.

              This was important as it allowed us to:

              * Match user group to module
              * Identify archetypes and device-access within each module
              * Eventually map out the default and master configuration so that the System can be used upon deployment

              From the research, we identified three primary archetypes within each module:

              * **Super Admin**: Needs overall visibility and ‘master controls’
              * **Manager**: Needs overall visibility and control within their domain
              * **End User**: Needs a straightforward and "noise-free" experience where only relevant tools are visible

              Depending on the module, the different archetypes have different devices to access the system - desktop or mobile. Additionally, we found out that the chain of command could be changed depending on the type of operations the users will undertake.

      - anchorId: creating_logic
        title: O4. Creating Logic & Access Controls
        showSectionTitle: true
        blocks:
          - content: |
              Once both research pieces were complete, we started working on the access control logic.

              Here, our goal was to strip away the complexity of manual permission-setting, which often leads to user error and security gaps. We worked closely with a small user group to determine the rights for each archetype, for each module. This was then carefully documented for the developers to hardcode the archetypes with the specific rights.

              While this seems like a rigid way to control rights, this was intentionally done to overcome an inherent problem of Access Control - the ‘Bootstrap Paradox’. It is a chicken-and-egg problem:

              * If there is no **Super Admin** defined, the System cannot propagate a chain of command
              * If modules have no roles with **owner-type rights**, the System itself cannot run as the modules would be unusable

              Our solution was a foundational/master logic within the backend that ensures the System is operational from the moment of deployment.

        showSectionTitle: true
        blocks:
          - content: |
              Access Controls is divided into 2 areas.

              **1. Role Management**

              This is a centralised dashboard in the form of a table that serves as the source of truth. Here, Super Admins can quickly verify which modules are active for each role and see a "Read-Only" summary of the hardcoded rights, maintaining clarity without allowing for accidental logic breaks.

              **2. Organizational Chart**

              This is a dynamic Org Chart where roles and permissions are localized into ‘Nodes’. Each node represents a specific role. When users log into the System, they are connected to a role by an external security system. Access Controls determines the module and archetype-specific permissions required for that user within the chain of command. This turns a complex backend database task into a simple, visual "drag-and-drop" exercise.
